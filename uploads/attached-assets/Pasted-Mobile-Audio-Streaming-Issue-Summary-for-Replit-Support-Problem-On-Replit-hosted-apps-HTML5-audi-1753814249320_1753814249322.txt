Mobile Audio Streaming Issue — Summary for Replit Support
Problem:
On Replit-hosted apps, HTML5 audio often spins indefinitely on mobile browsers (iOS Safari, Chrome Android) until the user manually seeks in the track. Desktop playback works normally.

Cause Indicators:

Likely combination of browser autoplay policy changes and server/CDN response headers not being optimal for streaming/seeking.

Behavior occurs even when autoplay is disabled and the user explicitly presses play.

Manual slider movement forces a partial seek request, which triggers proper buffering.

CDN may not be sending:

Content-Type: audio/mpeg (or correct MIME for format)

Accept-Ranges: bytes

Content-Range on partial requests

Access-Control-Allow-Origin: * for cross-domain use

Open [example audio URL] on a mobile browser directly (no app wrapper).

Workaround Implemented:
To unblock production users, the player was modified to include:

Universal gesture detection (touchstart, click, scroll, touchmove) to unlock audio on first interaction.

Hidden muted audio unlocker (silent MP3 played/paused once) to satisfy iOS gesture rules.

Auto-nudge seek (currentTime jump forward/back) to force buffering.

Cross-origin enabled (crossOrigin="anonymous") for CORS safety.

Detailed logging of readyState, networkState, and error codes for debugging.

Snippet 2: Auto-Nudge + Gesture Unlock
tsx
Copy
Edit
useEffect(() => {
  const unlockMobileAudio = async () => {
    if (!isMobile || mobileUnlocked) return;
    try {
      const unlocker = unlockerRef.current;
      if (unlocker) {
        unlocker.muted = true;
        await unlocker.play();
        unlocker.pause();
        setMobileUnlocked(true);
        console.log('Mobile audio unlocked via user gesture');
      }
    } catch (err) {
      console.error('Mobile audio unlock failed:', err);
    }
  };

  const handleUserGesture = (e: Event) => {
    console.log(`Mobile gesture detected: ${e.type}`);
    unlockMobileAudio();
  };

  // Listen for any user interaction
  ['touchstart', 'click', 'scroll', 'touchmove'].forEach(evt =>
    window.addEventListener(evt, handleUserGesture, { once: true })
  );

  return () => {
    ['touchstart', 'click', 'scroll', 'touchmove'].forEach(evt =>
      window.removeEventListener(evt, handleUserGesture)
    );
  };
}, [isMobile, mobileUnlocked]);

const autoNudge = async (audio: HTMLAudioElement) => {
  if (audio.readyState > 0 && audio.duration > 0) {
    const originalTime = audio.currentTime;
    audio.currentTime = 0.1; // Tiny seek forward
    setTimeout(() => {
      audio.currentTime = originalTime; // Return to original
      console.log('Auto-nudge applied to initialize audio streaming');
    }, 100);
  }
};
Why This Matters:
While the workaround is robust, it adds client complexity and delays playback on first interaction.
A server/CDN fix to send correct streaming headers and ensure mobile browsers can begin playback without a seek would make the workaround unnecessary.
